Microsoft Entra ID Front Door for Relayna Gateway

Future-feature design for placing OAuth2/OIDC identity in front of Relayna Gateway while preserving Relayna virtual keys as the gateway-owned policy and usage anchor.

Status: design proposal Compatibility boundary: v0.1.4 Production freeze baseline: v0.1.0 Scope: future auth and authorization feature

Decision

Provider traffic should require two independent credentials before Relayna Gateway forwards a request: a valid Microsoft Entra ID access token and a valid Relayna virtual key. Entra ID proves the caller identity and coarse gateway entitlement. The Relayna virtual key remains the durable Relayna identity for project attribution, model/provider policy, budget, rate limits, guardrails, and usage accounting.

Recommended request contract: use Authorization: Bearer <Entra JWT> for OAuth2/OIDC and send the Relayna virtual key in X-Relayna-Key: rk_live_.... Do not replace the current Authorization: Bearer rk_live_... behavior without an explicit migration plan.

Architecture

Relayna Gateway has two supported entrances. The direct entrance lets a client call the gateway after acquiring an Entra access token. The Apigee entrance adds an enterprise API-management edge that can also validate the token before the request reaches Relayna Gateway.

Request Flows

Direct Entra Entrance

  1. The app obtains an Entra access token for the Relayna Gateway API audience.
  2. The app calls Relayna Gateway with Authorization: Bearer <Entra JWT> and X-Relayna-Key: rk_live_....
  3. Relayna Gateway validates the JWT signature, issuer, audience, timestamps, tenant, version, and accepted algorithm.
  4. Relayna Gateway authorizes roles, scopes, and groups before virtual-key authentication.
  5. Existing Relayna key authentication, route matching, policy, rate limit, budget, guardrail, and usage logic runs.
  6. The gateway strips client credentials and forwards using only internal provider credentials.

Apigee Entrance

  1. The app obtains an Entra access token and sends the request through Apigee.
  2. Apigee validates the JWT with its VerifyJWT policy and can enforce edge controls such as quotas or threat protection.
  3. Default mode forwards the original JWT so Relayna Gateway revalidates it.
  4. Trusted mode may forward signed identity headers only over private networking and mTLS.
  5. Relayna Gateway verifies either the JWT or the Apigee trust proof before applying Relayna key and policy checks.
  6. Provider forwarding remains unchanged: no client JWT, no Relayna virtual key, only internal credentials.

Gateway Validation Rules

JWT validation must happen before any upstream provider call and before existing Relayna virtual-key policy permits the request. Validation should be fail-closed and observable without logging token material.

Validation Area Design Rule
Metadata and keys Resolve Entra OIDC metadata and JWKS from the configured tenant authority. Cache keys with rotation-aware refresh and retry. Reject tokens that cannot be validated against a trusted key.
Signature and algorithm Accept only configured asymmetric signing algorithms such as RS256. Reject unsigned tokens, algorithm confusion, missing kid, and unknown signing keys.
Issuer Single-tenant v1 design requires exact issuer match for the configured tenant. Multi-tenant support is future additive work and must require tenant allowlisting.
Audience Require aud to match the Relayna Gateway API application ID URI or client ID. Reject tokens minted for Microsoft Graph, Apigee, or any unrelated API.
Time claims Validate exp, nbf, and iat with a small configured clock-skew allowance. Expired and not-yet-valid tokens fail before policy evaluation.
Tenant and version Validate tid and token ver. The metadata endpoint must match token version expectations.
Identity context Extract only sanitized identity fields needed for authorization, telemetry, and audit, such as tenant ID, object ID, app ID, scopes, roles, and groups.

Middleware and Library Research

Microsoft positions MSAL as the client-side library family for acquiring and caching tokens. Relayna Gateway is the resource server in this design, so it should validate access tokens instead of using MSAL as the primary server-side enforcement mechanism. In .NET services, Microsoft recommends Microsoft.Identity.Web and JwtBearer middleware; in Relayna's Rust stack, the equivalent should be a gateway-owned verifier built from OIDC metadata, JWKS caching, JWT validation, and existing Axum/Tower/Pingora hooks.

Option Fit for Relayna Gateway Design Decision
MSAL client libraries Good for apps, CLIs, workers, or Studio clients that need to acquire Entra access tokens and refresh them. Document for callers, not for gateway-side validation. Clients should request a token for the Relayna Gateway API scope or app role.
Microsoft.Identity.Web / JwtBearer Best reference implementation for ASP.NET Core APIs. It validates bearer tokens and can customize audiences and validation parameters. Use as a behavior model, not as a direct dependency, because Relayna Gateway is Rust-based.
jsonwebtoken Rust crate Good low-level verifier for decoding and validating JWTs once the correct Entra JWK has been selected by kid. Recommended validation engine candidate. Add with an explicit crypto backend and wrap it behind a gateway-core trait.
openidconnect Rust crate Useful for OIDC discovery types, provider metadata, JWKS structures, and OIDC claim concepts. Candidate for discovery and typed metadata. Validate access-token needs carefully because many OIDC helpers focus on ID tokens and relying-party flows.
oauth2 Rust crate Useful when Relayna itself must acquire tokens for outbound calls or on-behalf-of flows. Not required for inbound bearer JWT validation in the first feature slice.
Custom Axum/Tower middleware Fits the control-plane API because Axum supports middleware functions that inspect headers and short-circuit unauthorized requests. Use for Axum-owned routes that need Entra enforcement. Insert sanitized identity context into request extensions for handlers.
Pingora proxy hook Required for provider traffic because high-throughput proxy routes are owned by gateway-proxy, not only Axum. Call the same gateway-core verifier before virtual-key auth, policy, budget, and upstream credential injection.
Apigee VerifyJWT policy Strong edge validation option that can validate signatures, timestamps, selected claims, and JWKS keys before traffic reaches Relayna Gateway. Use at the edge, but keep Relayna Gateway revalidation as the default trust model.

Recommended Rust Shape

  • Add a framework-agnostic EntraJwtVerifier in gateway-core with inputs for token string, expected audience, tenant, issuer mode, required roles/scopes, and accepted groups.
  • Use the existing reqwest dependency to fetch the Entra OIDC discovery document and JWKS. Cache metadata and keys with TTLs, refresh on unknown kid, and fail closed if refresh cannot validate the token.
  • Use jsonwebtoken or a similarly audited JWT crate for signature and registered-claim validation. Do not use unsafe decode paths except for reading the untrusted header to choose a JWK by kid.
  • Represent validated identity as a small immutable context containing tid, oid or app identity, azp/appid, roles, scopes, groups, and token mode. Redact raw tokens from logs.
  • Keep authorization rules data-driven through config or policy tables later, but start with static config for tenant, audience, required invoke role/scope, and accepted groups.
  • For group overage claims, fail closed in v1 and document that Microsoft Graph lookup is a later additive feature requiring Graph permissions, cache policy, and failure semantics.

Authorization Model

Entra authorization gates who may connect to Relayna Gateway. Relayna policy still decides what an accepted caller may do with AI traffic.

App Roles and Scopes

  • gateway.invoke permits provider traffic entry.
  • gateway.admin.read permits future read-only admin API access.
  • gateway.admin.write permits future mutating admin API access.
  • Client-credentials callers should use app roles. Delegated user callers may use scopes.

Groups

  • Group claims may map to Relayna projects, inherited policy layers, or gateway access tiers.
  • Group checks should be allowlist based, not broad tenant membership checks.
  • When Entra emits group overage indicators, Relayna should fail closed until Graph lookup or a dedicated group-resolution service is implemented.
Policy ordering: Entra role, scope, and group checks only decide whether the caller may enter the gateway. Existing Relayna virtual key policy remains authoritative for routes, models, providers, streaming, tools, tasks, budgets, rate limits, guardrails, and usage.

Apigee Trust Modes

Mode Gateway Behavior When to Use
JWT revalidation Apigee validates the Entra JWT and forwards the original JWT. Relayna Gateway validates the same JWT again before virtual-key auth and policy. Recommended default. It gives defense in depth and keeps Relayna Gateway independently secure if Apigee policy is changed or bypassed.
Trusted internal headers Apigee validates Entra ID and forwards signed identity headers. Relayna Gateway accepts them only after verifying mTLS/private-network controls and a header signature or equivalent trust proof. Use only for tightly controlled private deployments where duplicate JWT validation is intentionally avoided and the Apigee trust boundary is enforced.
Trusted-header mode must never be enabled on a public listener. A direct caller must not be able to forge Apigee identity headers.

Compatibility and Implementation Notes

  • Latest release tag observed during planning: v0.1.4.
  • Production freeze baseline remains v0.1.0.
  • Future implementation touches frozen surfaces: authentication behavior, public request headers, policy decisions, proxy forwarding, telemetry fields, config/env vars, and freeze perimeter tests.
  • Use additive configuration and headers for the first implementation. Do not silently reinterpret Authorization: Bearer rk_live_... as an Entra token.
  • Add stable JWT-specific error codes before implementation, for example missing JWT, malformed JWT, invalid JWT, expired JWT, invalid audience, invalid issuer, and insufficient Entra authorization.
  • Update tests/freeze-v0.1.0-perimeter.test.mjs when the future feature adds public error codes, config names, or request-header contracts.
  • Run the gateway verification stack for future runtime changes: cargo fmt --all --check, cargo clippy --workspace --all-targets --all-features -- -D warnings, and cargo test --workspace --all-features.

Future Implementation Test Plan

  • Valid Entra JWT plus valid Relayna key forwards successfully.
  • Missing JWT, invalid signature, wrong audience, wrong issuer, expired token, and missing required scope or role fail before upstream forwarding.
  • Valid JWT with invalid, disabled, revoked, or expired Relayna key still fails under existing virtual-key rules.
  • Apigee revalidation mode rejects tampered, missing, or wrong-audience JWTs.
  • Trusted Apigee-header mode rejects requests without required mTLS, private network, and signed-header trust proof.
  • Provider calls never receive client Entra JWTs or Relayna virtual keys.
  • Usage events include Relayna key/project attribution and only sanitized Entra identity fields selected by the implementation.
